## 2    Problem Statement:  Task Board
Before starting this assignment, familiarize yourself with what Jira and Kanban boards
are:
-  What is Jira? https://www.atlassian.com/software/jira/guides/getting-started/
introduction
-  What is a Kanban Board? https://www.atlassian.com/agile/kanban/boards
## 2.1    Overview
A project management and issue-tracking application inspired by Jira.  It enables teams
to plan, track, and manage work using boards, issues, workflows, and more.
## 2.2    Project Goals
-  Provide a Kanban-style task tracking system
-  Support projects, boards, issues, and workflows
-  Enable team collaboration and role-based access
-  Maintain auditability and traceability of changes
## 2.3    Functional Requirements
## 2.3.1    Authentication, Authorization & User Management
User Registration and Login
-  Email + password authentication
-  Passwords must be hashed
## Session Management
-  Use JWT (JSON Web Tokens) for authentication
-  Access tokens stored in HTTP-only cookies
-  Implement refresh token mechanism for session renewal
-  Logout must invalidate the refresh token
## User Profile
## •  Name
## •  Email
-  Avatar upload and display

Role-Based Access Control
There are two levels of roles:
## Global Roles:
-  Global Admin:  Can create projects,  manage all users,  and assign project-level
roles
Project-Level Roles  (users can have different roles across different projects):
-  Project Admin:  Full access to project settings, can manage project members and
edit workflows
-  Project Member:  Create and edit tasks, participate in projects
-  Project Viewer:  Read-only access to project and its tasks
Only Global Admins and Project Admins can assign roles within a project.
## 2.3.2    Projects
## Project Management
-  Create, update, and archive projects
-  Assign users to projects
-  Project-level permissions
## Project Metadata
## •  Name
## •  Description
-  Creation and modification timestamps
2.3.3    Boards (Kanban)
## Board Structure
-  Each project has one or more boards
-  Each board has its own workflow (column configuration)
-  Each column corresponds to a workflow status
-  Default columns:  To Do, In Progress, Review, Done (customizable per board)
-  Drag-and-drop tasks between columns
-  Column-level  WIP  (Work  In  Progress)  limits  must  be  enforced  —  invalid  moves
should be blocked, not just warned
## Board Features
-  Add, rename, reorder, and delete columns
-  Visual feedback during drag operations
-  Only Project Admins can modify board workflows (add/rename/delete columns)

## 2.3.4    Issues / Tasks
Task Types (Hierarchical)
-  Story:  Parent  item  that  contains  related  Tasks  and  Bugs  (not  directly  movable
across columns)
-  Task:  Standard work item, can be linked to a Story as a child
-  Bug:  Issue to be fixed, can be linked to a Story as a child
Parent-child relationships must be enforced.  A Story’s status can be automatically de-
rived from its children’s statuses, or manually set.  If manually set, the Story status must
be consistent with its children’s states.
## Task Fields
## •  Title
## •  Description
-  Status (based on workflow)
-  Priority (Low, Medium, High, Critical)
-  Assignee (team member assigned to task)
-  Reporter (user who created the task)
-  Due date
## Task Operations
-  Create, edit, and delete tasks
-  Move tasks between columns
-  Change task status, priority, and assignment
-  Task details view with full information
## 2.3.5    Issue Lifecycle & Workflow
-  Configurable workflows per board
-  Valid status transitions (e.g., To Do→ In Progress→ Done)
-  Invalid status transitions must be blocked
Audit Trail (Mandatory)
The following events must be logged for each task:
-  Status changes (with old and new status)
-  Assignee changes (with old and new assignee)
-  Comments added/edited/deleted

-  Comment threads on tasks

## 2.3.7    Notifications

Polling-based  notification  fetching  is  acceptable;  real-time  (WebSocket)  delivery  is  op-
